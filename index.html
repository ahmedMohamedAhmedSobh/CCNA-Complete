Network types
  • LAN — Local Area Network: small geographic area (home, building, office)
  • WAN — Wide Area Network: connects multiple LANs; spans cities/countries; owned by ISP
  • WLAN — Wireless LAN (802.11)
  • MAN — Metropolitan Area Network (city-wide)
  • SAN — Storage Area Network (storage devices)
  • SOHO — Small Office/Home Office: small LAN, usually <10 devices
Network components
  • End devices: PCs, phones, printers, servers, IP cameras, IoT sensors
  • Intermediary devices: routers, switches, multilayer switches, firewalls, wireless APs, hubs (legacy)
  • Media: copper UTP, fiber optic, wireless RF
  • NIC: connects device to network; has a MAC address burned in (BIA)
Physical vs Logical topology
  • Physical topology: actual layout of cables, rooms, buildings — what you can see and touch
  • Logical topology: how data flows — IP addressing, VLAN design, routing paths
  • Both are required in professional network documentation
Internet · Intranet · Extranet
  • Internet: global interconnected public network; no single owner
  • Intranet: private internal network; accessible by employees only
  • Extranet: allows limited controlled external access to intranet (partners, suppliers, customers)

Delivery options
  • Unicast: 1 sender → 1 specific receiver (1-to-1). Example: web browsing, SSH
  • Multicast: 1 sender → subscribed group (1-to-many). Example: video streaming, OSPF (224.0.0.5), EIGRP (224.0.0.10)
  • Broadcast: 1 sender → ALL devices on LAN (1-to-all). IPv4 ONLY — IPv6 has NO broadcast. Example: ARP, DHCP Discover
  • Anycast: 1 sender → nearest member of a group. IPv6 + DNS load balancing

Encapsulation / De-encapsulation
  • Encapsulation: data travels DOWN layers at sender — each layer adds its header (L2 also adds trailer)
  • De-encapsulation: data travels UP layers at receiver — each layer strips its header
  • Each layer treats the upper layer's PDU as its payload — it does not read the content
  • Same-layer communication: L3 header on sender is interpreted only by L3 on receiver
Module 4 — Physical Layer
ITN — Module 4
Exam focus: 3 physical layer functions, UTP categories + 100m limit, T568A vs T568B pin colors, cable types (straight/crossover/rollover + when to use), fiber SMF vs MMF (color, distance, light source), full vs half duplex.
Physical layer — 3 functions
  • Physical components: NICs, ports, connectors, cables, transceivers (hardware)
  • Encoding: converts bit stream into code (Manchester, 4B/5B, 8B/10B, 64B/66B). Receiver must know the encoding to decode.
  • Signaling: represents bits as signals — voltage change (copper), light pulse (fiber), radio wave (wireless)
Standards: IEEE, ISO, ANSI, TIA, EIA, ITU-T define connectors, media types, speeds, distances
Full duplex vs Half duplex
  • Half-duplex: send OR receive, not simultaneously. Walkie-talkie. Uses CSMA/CD (wired) or CSMA/CA (wireless). Hubs force half-duplex on all devices.
  • Full-duplex: send AND receive simultaneously. Telephone. No collisions possible. Switches support full-duplex per port.
  • Duplex mismatch: one side full, other half → CRC errors, slow performance, retransmissions
  • Auto-MDIX: switches automatically adjust for wrong cable type

Module 6 — Data Link Layer
ITN — Module 6
Exam focus: LLC vs MAC sublayers, CSMA/CD vs CSMA/CA, frame responsibilities, physical/logical topologies, WAN topologies (point-to-point, hub-spoke, mesh).
Two IEEE 802 sublayers
  • LLC (Logical Link Control) — 802.2: communicates with upper Network layer; identifies which L3 protocol is in the frame (IPv4=0x0800, IPv6=0x86DD, ARP=0x0806); error checking
  • MAC (Media Access Control) — 802.3/802.11: data encapsulation (framing); physical addressing (MAC addresses); media access control — determines when to transmit
Data Link responsibilities
  • Framing: encapsulates L3 packet into a frame for THIS hop
  • Physical addressing: adds source + destination MAC to frame
  • Media access: controls who transmits when (CSMA/CD, CSMA/CA, Token)
  • Error detection: FCS in trailer lets receiver detect (NOT correct) corrupted frames
  • Hop-to-hop: MAC changes at every router hop; IP stays same end-to-end
Media access control methods
MethodUsed byHow it works
CSMA/CD — Collision DetectionLegacy wired Ethernet (hubs, half-duplex)Listen before transmit (CS). Multiple access (MA). If collision: all stop, send 32-bit jam signal, each waits random backoff, retry. Full-duplex Ethernet does NOT use CSMA/CD — no collisions possible.
CSMA/CA — Collision AvoidanceWireless 802.11Listen before transmit. Even if clear, wait random backoff. Optionally RTS/CTS (Request/Clear to Send) to reserve channel. Cannot detect collision while transmitting (would interfere), so avoids instead.
Token passingToken Ring 802.5, FDDI (legacy)Token circulates; only holder can transmit. No collisions. Deterministic. Rarely used today.
LAN topologies
  • Star (extended star): all connect to central switch. Most common today. Easy management.
  • Bus: all on single coax cable. Legacy 10BASE2/10BASE5. All share collision domain.
  • Ring: Token Ring (legacy 802.5). Token passed around.
  • Mesh: multiple connections between devices. Used in WAN for redundancy.
WAN topologies
  • Point-to-point: dedicated link between 2 sites. Simplest, most common WAN topology.
  • Hub-and-spoke: central hub, multiple spoke branches (star over WAN). Efficient, but hub is single point of failure.
  • Full mesh: every site connects to every other. Maximum redundancy. Cost = n(n-1)/2 links.
  • Partial mesh: some redundant connections. Balance between cost and redundancy.

MAC address structure
  • 48 bits = 6 bytes = 12 hex digits
  • First 3 bytes (24 bits) = OUI (Organizationally Unique Identifier) — assigned by IEEE to vendor
  • Last 3 bytes (24 bits) = unique device ID — assigned by vendor
  • Formats: AA:BB:CC:DD:EE:FF or AA-BB-CC-DD-EE-FF or AABB.CCDD.EEFF (Cisco notation)
  • BIA (Burned-In Address): stored in NIC ROM at factory
  • Can be overridden in software = locally administered address (LAA)
  • LSB of first byte: 0 = unicast, 1 = multicast/broadcast
Special MAC addresses
  • FF:FF:FF:FF:FF:FF — Ethernet broadcast (all devices on same segment)
  • 01:00:5E:xx:xx:xx — IPv4 multicast (last 23 bits from multicast group IP)
  • 33:33:xx:xx:xx:xx — IPv6 multicast
  • All others — unicast

Collision domain vs Broadcast domain
  • Collision domain: area where 2 devices can collide. Each switch port = separate collision domain. Hub = ONE big collision domain.
  • Broadcast domain: area where broadcasts are forwarded. Entire switch = ONE broadcast domain (unless VLANs used). Routers separate broadcast domains.
  • Summary: Switch breaks collision domains but NOT broadcast domains. Router breaks BOTH.
ARP — Address Resolution Protocol
  • Resolves IPv4 address → MAC address on local network
  • ARP Request: broadcast FF:FF:FF:FF:FF:FF — "Who has 192.168.1.5? Tell 192.168.1.1"
  • ARP Reply: unicast — "I have 192.168.1.5. My MAC is AA:BB:CC:DD:EE:FF"
  • Cached in ARP table (expires). arp -a on PC, show arp on Cisco router
  • ARP spoofing: attacker sends fake replies → poisons cache → MitM attack
  • Mitigation: Dynamic ARP Inspection (DAI) on switches

IP 3 key characteristics
  • Connectionless: no session established before sending. No handshake at L3. Each packet treated independently.
  • Best-effort: no guaranteed delivery, no acknowledgment, no error recovery at L3. TCP at L4 handles reliability if needed.
  • Media independent: IP works over ANY L2 technology (Ethernet, WiFi, fiber, serial, PPP) without change. MTU varies per media — IP fragments if necessary.

Module 9 — Address Resolution
ITN — Module 9
Exam focus: Same-network vs remote-network communication — which MAC changes, which IP stays the same. IPv6 NDP messages (RS/RA/NS/NA) and their ICMPv6 type numbers.
Same-network communication (local)
  • Host checks: is dest IP in my subnet? YES
  • Checks ARP cache for dest IP→MAC mapping
  • If not cached: ARP broadcast to get dest MAC
  • Frame: Dest MAC = receiver's MAC, Src MAC = sender's MAC
  • Packet: Dest IP = receiver's IP, Src IP = sender's IP
Remote-network communication (different subnet)
  • Host checks: is dest IP in my subnet? NO
  • Must send to default gateway (router)
  • Checks ARP cache for default gateway's MAC
  • If not cached: ARP broadcast to get router's MAC
  • Frame: Dest MAC = ROUTER's MAC, Src MAC = sender's MAC
  • Packet: Dest IP = FINAL destination, Src IP = sender's IP
  • KEY RULE: MAC changes at every hop. IP stays same end-to-end.

Module 12 — IPv6 Addressing
ITN — Module 12

Why IPv6?
  • IPv4: 2^32 = ~4.3 billion addresses — exhausted at IANA level in 2011
  • IPv6: 2^128 = 340 undecillion — effectively unlimited
  • No NAT needed — every device can be globally reachable
  • No broadcast — uses multicast and anycast
  • Simplified fixed 40-byte header (vs variable IPv4 header)
  • Built-in IPsec (authentication + encryption natively supported)
  • SLAAC: hosts auto-configure without a DHCP server
IPv6 abbreviation rules — 2 rules
  • Rule 1: Omit LEADING zeros in each 16-bit hextet. 0DB8 → DB8, 0000 → 0, 0001 → 1
  • Rule 2: Replace ONE consecutive sequence of all-zero hextets with :: (double colon). Can only use :: ONCE per address.
  • Full: 2001:0DB8:0000:0000:0000:0000:0000:0001
  • After Rule 1: 2001:DB8:0:0:0:0:0:1
  • After Rule 2: 2001:DB8::1
  • Loopback: 0000:...:0001 → ::1
  • Unspecified: 0000:...:0000 → ::

TCP 3-way handshake (connection establishment)
  • Step 1: Client → Server: SYN (Synchronize) — "I want to connect, my starting seq# is X"
  • Step 2: Server → Client: SYN-ACK — "OK, my starting seq# is Y, I acknowledge your X+1"
  • Step 3: Client → Server: ACK — "I acknowledge your Y+1"
  • Connection established — data can flow
TCP connection termination (4-way)
  • Step 1: Client → Server: FIN (want to close)
  • Step 2: Server → Client: ACK
  • Step 3: Server → Client: FIN (server closes its side)
  • Step 4: Client → Server: ACK
  • Connection fully closed

TCP flow control — window size
  • Window size: max bytes sender can transmit before receiving an ACK
  • Determined by receiver's available buffer space
  • Large window = more throughput (less waiting)
  • Small window = more ACKs = more overhead
  • Sliding window: window can grow/shrink dynamically
  • TCP congestion control: reduces window size if congestion detected (packet loss)

  • Two browser tabs = two different source port numbers → two different sockets

Module 2 — Switching Concepts
SRWE — Module 2
Exam focus: MAC table (CAM table) operation, frame flooding vs forwarding, collision domain vs broadcast domain (per switch vs per VLAN), store-and-forward vs cut-through again.
How a switch builds its MAC address table
ActionDescription
LearnWhen ANY frame arrives, switch reads SOURCE MAC and records: MAC → port → VLAN in CAM table. Default aging time = 300 seconds.
Forward (known)If destination MAC is in table → send ONLY to that port. Microsegmentation.
Flood (unknown unicast)If destination MAC NOT in table → send to ALL ports in same VLAN except the source port.
Flood (broadcast)Destination = FF:FF:FF:FF:FF:FF → flood to ALL ports in same VLAN except source.
FilterSource and destination on same port → drop (no need to forward).
Collision domain vs Broadcast domain
  • Collision domain: area where 2 transmissions collide. Each switch port = 1 collision domain. Hub = 1 big domain.
  • Broadcast domain: area where broadcasts propagate. All switch ports in same VLAN = 1 domain. Router = boundary.
  • VLANs: each VLAN = separate broadcast domain. Switch with 3 VLANs = 3 broadcast domains.
  • Switches break collision domains. Routers + VLANs break broadcast domains.
Switch forwarding methods (review)
  • Store-and-Forward: waits for entire frame + FCS check. Drops bad frames. Default Cisco behavior.
  • Cut-through: reads 6 bytes (dest MAC), forwards immediately. Low latency, no error check.
  • Fragment-free: reads 64 bytes. Catches collision fragments (runts). Compromise.
  • Cisco Catalyst switches default to store-and-forward.

Why STP? — The L2 loop problem

Module 8 — SLAAC & DHCPv6
SRWE — Module 8

SLAAC — Stateless Address Autoconfiguration
  • Host sends Router Solicitation (RS) to FF02::2 (all routers)
  • Router replies with Router Advertisement (RA): prefix /64, gateway, M/O flags
  • Host creates its own address: RA prefix + EUI-64 interface ID (or random)
  • Host performs DAD (Duplicate Address Detection) to verify address is unique
  • No DHCPv6 server needed — fully automatic
  • RA sent periodically even without RS (every 200 seconds by default)
RA flags — M and O
  • M flag (Managed Address Config): 0 = use SLAAC, 1 = use stateful DHCPv6 for address
  • O flag (Other Config): 0 = no DHCPv6, 1 = use stateless DHCPv6 for DNS/options only
  • M=0, O=0 → SLAAC only (no DHCPv6)
  • M=0, O=1 → SLAAC + stateless DHCPv6 (for DNS)
  • M=1, O=0 → stateful DHCPv6 (full address from server)
  • M=1, O=1 → stateful DHCPv6 for address + options
DAD — Duplicate Address Detection
  • Before using a new IPv6 address (SLAAC or static), host sends Neighbor Solicitation (NS) with source = unspecified (::)
  • Target = the candidate IPv6 address
  • If Neighbor Advertisement (NA) received: address is already in use → conflict
  • If no reply: address is unique → host assigns it to interface
  • DAD applies to ALL IPv6 addresses (GUA and LLA)

Why FHRP?
  • Problem: hosts are configured with ONE default gateway IP. If that router fails → all hosts on subnet lose connectivity. Static default gateway = single point of failure.
  • Solution: FHRP creates a virtual IP (VIP) and virtual MAC shared by multiple routers. Hosts use VIP as gateway. If primary fails, backup takes over seamlessly.
  • Hosts don't know or care which physical router is actually handling traffic — they only see the virtual IP.

AAA Framework
  • Authentication: verifies WHO you are (username + password, certificate, biometric)
  • Authorization: determines WHAT you're allowed to do (privileges, commands, resources)
  • Accounting: records WHAT you did (logs commands, session time, data transferred)

  • EAP (Extensible Authentication Protocol): carries authentication messages
  • PEAP, EAP-TLS: common EAP methods
  • Result: only authenticated devices get network access, regardless of physical port

Module 14 — Routing Concepts
SRWE — Module 14

What a router does — two core functions
FunctionDescription
Path determinationThe router examines the destination IP in every incoming packet and uses the routing table to determine the best path to forward it. This happens at Layer 3.
Packet switching (forwarding)After the path is chosen, the router strips the old Layer 2 frame header, builds a new one appropriate for the outgoing interface, and sends the packet out. Layer 2 changes every hop; Layer 3 stays the same.
Router memory types
  • ROM — Read-Only Memory. Holds the bootstrap program and the POST (Power-On Self Test). The router loads from here on boot and uses it to find IOS in Flash.
  • Flash — Storage for the IOS image(s). Non-volatile. Survives reboot. Can hold multiple IOS versions.
  • RAM — Random Access Memory. Holds the running-config, routing table, ARP cache, and the packet buffer. Volatile — contents lost on reboot.
  • NVRAM — Non-Volatile RAM. Holds the startup-config (saved configuration). Survives reboot. Small and fast.
Router boot sequence
  • Step 1: POST runs from ROM — tests CPU, RAM, interfaces
  • Step 2: Bootstrap in ROM loads — finds and loads IOS from Flash
  • Step 3: IOS loads into RAM from Flash
  • Step 4: IOS copies startup-config from NVRAM into RAM as running-config
  • Step 5: If no startup-config exists → router enters initial setup dialog
  • show version — shows IOS version, uptime, Flash/RAM sizes, config register

Routing table lookup — how the router decides
  • Step 1: Router receives a packet, reads the destination IP address.
  • Step 2: Searches routing table for the longest prefix match (most specific route = most bits matching). A /28 beats a /24 beats a /0.
  • Step 3: If a match is found → forward the packet out the specified interface to the next-hop IP.
  • Step 4: If no match and a default route (S*) exists → forward using the default route.
  • Step 5: If no match and no default route → drop the packet and send ICMP Destination Unreachable back to source.
  • Tie-breaking: Same prefix length, same AD → use metric (cost). Same metric → load balance (equal-cost load balancing, ECMP).
Packet forwarding mechanisms
  • Process switching (legacy): every packet triggers a CPU lookup of the routing table. Very slow. Not used today.
  • Fast switching (legacy): first packet uses CPU, result cached. Subsequent packets to same dest use cache. Faster but still mostly CPU.
  • Cisco Express Forwarding (CEF): current standard. Two tables pre-built in hardware — FIB (Forwarding Information Base, based on routing table) and Adjacency Table (pre-built L2 headers). No CPU per-packet. Line-rate forwarding.

What happens to a static route when the outgoing interface goes down?

Why OSPF — link-state routing
  • OSPF (Open Shortest Path First) is a link-state routing protocol — routers share information about their directly connected links, not just distances.
  • Each router builds a complete map of the entire network called the LSDB (Link-State Database).
  • Every router runs the Dijkstra SPF (Shortest Path First) algorithm independently on its LSDB to compute the shortest path to every destination.
  • OSPF is classless — supports VLSM and CIDR. Metric is cost (based on bandwidth). Lower cost = better path.
  • OSPF is an open standard (RFC 2328) — runs on any vendor's equipment. Administrative distance = 110.
  • OSPFv2 = for IPv4. OSPFv3 = for IPv6 (and can also carry IPv4 with address families).
OSPF packet types — 5 types
TypeNamePurpose
1HelloDiscover neighbors, establish/maintain adjacency, elect DR/BDR. Sent every 10 sec on broadcast networks, 30 sec on NBMA.
2DBD (Database Description)Summary of the LSDB sent during neighbor initialization. Used to synchronize databases.
3LSR (Link-State Request)Requests specific LSA entries that the router doesn't have after comparing DBDs.
4LSU (Link-State Update)Contains one or more LSAs. The actual routing information exchange.
5LSAck (Link-State Acknowledgment)Confirms receipt of LSAs (reliable delivery of LSAs).
OSPF cost — metric calculation
  • OSPF cost = Reference bandwidth / Interface bandwidth
  • Default reference bandwidth = 100 Mbps (100,000,000 bps)
  • FastEthernet (100 Mbps): 100/100 = cost 1
  • Gigabit Ethernet (1 Gbps): 100/1000 = cost 0.1 → rounded to 1
  • Problem: Fast Ethernet and GigE both get cost 1 — not differentiated!
  • Fix: Change reference bandwidth on ALL routers: auto-cost reference-bandwidth 10000 (for 10G networks)
  • Or manually set: interface g0/0 → ip ospf cost 10
  • Lower cost = preferred path (total cost = sum of outgoing interface costs on path)
OSPF neighbor states — 8 states in order
StateWhat is happening
DownNo Hello packets received from this neighbor. Initial state.
InitHello received from neighbor but our Router ID not in it yet. Neighbor doesn't know about us yet.
2-WayOur Router ID IS in neighbor's Hello. Bidirectional communication established. DR/BDR election happens here on broadcast networks.
ExStartDetermine Master/Slave relationship for DBD exchange. Higher Router ID = Master.
ExchangeRouters exchange DBD (Database Description) packets — summaries of their LSDBs.

On broadcast networks, only DR and BDR form Full adjacency with all other routers. DROther routers only reach 2-Way state with each other.
DR and BDR — Designated Router
  • Problem: on a broadcast multi-access network (Ethernet), N routers would form N(N-1)/2 adjacencies — too many LSA floods.
  • Solution: elect ONE Designated Router (DR) and one Backup DR (BDR). All other routers (DROther) form adjacency with DR and BDR only.
  • DR receives LSAs from all DROthers and floods to all others. BDR monitors DR and takes over if it fails.
  • DR election: highest OSPF priority wins (default = 1, range 0–255). Priority 0 = never becomes DR/BDR. Tiebreaker = highest Router ID.
  • Election is NOT preemptive — once elected, DR keeps role even if better router joins. Changing priority requires OSPF process reset.
  • Multicast addresses: DROther → DR/BDR = 224.0.0.6. DR → all OSPF routers = 224.0.0.5.
OSPF Router ID
  • Every OSPF router needs a unique 32-bit Router ID (looks like an IP address)
  • How it's chosen (in order of priority):
  • 1. Manually configured: router ospf 1 → router-id 1.1.1.1 (preferred — stable)
  • 2. Highest loopback interface IP (loopbacks are always up)
  • 3. Highest active physical interface IP (can change if interface goes down)
  • Changing the router-id requires: clear ip ospf process
  • Verify: show ip ospf — shows Router ID at top

Hashing — integrity verification
  • Hash = a one-way mathematical function that produces a fixed-length "fingerprint" of data
  • Even ONE bit change in data = completely different hash output (avalanche effect)
  • Cannot reverse a hash to get original data
  • MD5 (128-bit): fast but BROKEN — do not use for security
  • SHA-1 (160-bit): deprecated — BROKEN
  • SHA-256 / SHA-3: current standard — secure
  • HMAC (Hash-based Message Authentication Code): combines hash with a secret key — provides both integrity AND authentication
  • Used in: password storage, digital signatures, file integrity verification, TLS
PKI — Public Key Infrastructure
  • PKI manages digital certificates that bind a public key to an identity
  • CA (Certificate Authority): trusted third party that issues and signs digital certificates
  • Digital certificate contains: subject's public key, subject identity, CA's digital signature, validity period
  • Certificate chain: Root CA → Intermediate CA → End-entity certificate
  • Used in: HTTPS, VPNs, code signing, email encryption (S/MIME)
  • CRL (Certificate Revocation List): list of revoked certificates
  • OCSP (Online Certificate Status Protocol): real-time revocation check (faster than CRL)
Defense-in-depth — layered security strategy
  • No single security control is sufficient — use MULTIPLE overlapping layers. If one fails, others still protect.
  • Physical security: lock the data center, badge access, cameras
  • Network security: firewalls, IPS/IDS, ACLs, network segmentation (DMZ, VLANs)
  • Endpoint security: antivirus, EDR (Endpoint Detection and Response), host firewall, patch management
  • Application security: WAF (Web Application Firewall), secure coding practices, input validation
  • Data security: encryption at rest + in transit, DLP (Data Loss Prevention), access controls
  • User security: security awareness training, MFA, least privilege, zero trust

What ACLs are and what they do
  • ACL (Access Control List) = a sequential list of permit/deny rules (ACEs — Access Control Entries) applied to router interfaces
  • ACLs examine packet header fields to decide: PERMIT (allow through) or DENY (drop)
  • ACLs process rules sequentially — first match wins, no further rules checked
  • IMPLICIT DENY: every ACL has an invisible "deny any" at the END. If no rule matches, packet is DROPPED. This is the most common mistake — forgetting that a new ACL blocks everything not explicitly permitted.
  • ACLs can filter: incoming traffic (inbound), outgoing traffic (outbound), or both on an interface
  • ACLs are created globally then APPLIED to an interface in a direction

Wildcard masks in ACLs
  • Wildcard mask tells the ACL which bits of the IP address MUST match (0 = must match) and which can be ANYTHING (1 = don't care)
  • 0.0.0.0 = match ALL bits exactly (this one specific host)
  • 0.0.0.255 = match first 24 bits, last 8 bits can be anything (entire /24 network)
  • 255.255.255.255 = match nothing — all bits are don't-care (matches any host) — use keyword any instead
  • Shortcut keywords: host 10.1.1.1 = 10.1.1.1 0.0.0.0. any = 0.0.0.0 255.255.255.255
  • Non-contiguous wildcard: can match specific ranges (e.g., 0.0.0.15 matches last 4 bits only). Powerful but complex.
Numbered ACL — configuration

  • Standard deny: access-list 10 deny host 192.168.1.100

  • Extended deny ICMP: access-list 110 deny icmp any any
  • Apply to interface: interface g0/0 → ip access-group 10 in
  • Apply to VTY (SSH/Telnet): line vty 0 15 → access-class 10 in
  • Remove from interface: no ip access-group 10 in
  • Delete ACL: no access-list 10
Named ACL — configuration (preferred)
  • Create: ip access-list standard BLOCK-HOSTS
  • Add rules: 10 deny host 192.168.1.100
  • Add rules: 20 permit 192.168.1.0 0.0.0.255
  • Create extended: ip access-list extended FILTER-WEB
  • Rules: 10 permit tcp 10.0.0.0 0.255.255.255 any eq 443
  • Apply: interface g0/0 → ip access-group FILTER-WEB in
  • Advantages over numbered: descriptive names, can add/delete individual sequence numbers, can insert rules between existing rules

Why NAT — the problem it solves
  • IPv4 addresses are exhausted. Private addresses (RFC 1918) cannot be routed on the internet.
  • NAT (Network Address Translation) translates private addresses → public addresses at the border router, allowing multiple devices to share one or a few public IPs.
  • A home with 10 devices can share ONE public IP from the ISP using PAT.
  • NAT also provides a layer of security by hiding internal addressing from the internet.

MPLS — Multiprotocol Label Switching
  • MPLS uses SHORT FIXED-LENGTH LABELS (not long IP addresses) to forward packets — much faster switching decisions
  • Customer traffic enters MPLS network at PE (Provider Edge) router → labeled → forwarded through P (Provider) core routers using label → label removed at far-end PE
  • Traffic Engineering: MPLS can route traffic along specific paths regardless of IP routing — useful for QoS and redundancy
  • MPLS VPN: provides any-to-any connectivity between customer sites over SP backbone. Customer sites see each other as directly connected over a private network.
  • Advantages: high performance, traffic engineering, supports multiple protocols, QoS
  • Disadvantages: expensive, managed by SP (less control), not end-to-end encrypted by default
SD-WAN — Software-Defined WAN
  • SD-WAN virtualizes WAN connections — separates the control plane from the data plane
  • Can use ANY combination of WAN links (MPLS + broadband + 4G) and intelligently route traffic across them
  • Central controller manages all WAN routers — policy-based routing, application awareness
  • Application-aware routing: send video calls over low-latency link, bulk file transfers over cheaper broadband
  • Advantages: lower cost (replace expensive MPLS with cheaper broadband), centralized management, application visibility, automatic failover
  • Use case: enterprise with many branches connecting to cloud + HQ

Why VPN
  • VPN (Virtual Private Network) creates an encrypted tunnel over a public network (internet), allowing private communication between two endpoints.
  • Benefits: confidentiality (encryption), integrity (hashing), authentication (verifies endpoints), access control
  • Replaces expensive dedicated leased lines with secure connections over cheap internet links

IPsec — the VPN framework
  • IPsec is a SUITE of protocols that provides authentication, integrity, and confidentiality for IP packets
  • Two main protocols:
  • AH (Authentication Header): provides authentication + integrity. Does NOT encrypt. Just verifies packets weren't tampered and came from legitimate source. NOT enough for confidentiality.
  • ESP (Encapsulating Security Payload): provides authentication + integrity + ENCRYPTION. Used in virtually all VPNs. Most important.
  • Two modes:
  • Transport mode: only payload encrypted. Original IP header left intact. Used for host-to-host communication.
  • Tunnel mode: ENTIRE original IP packet is encrypted and encapsulated inside a NEW IP packet. Most common for VPNs. Hides original source and destination IPs.
IKE — Internet Key Exchange
PhaseWhat happens
IKE Phase 1Establish a secure, authenticated channel between the two VPN endpoints. Negotiate IKE policy (encryption algo, hash, DH group, authentication method). Result: IKE SA (Security Association) — a management tunnel.
IKE Phase 2Using the secure Phase 1 channel, negotiate the actual IPsec SA — the parameters for encrypting user data (encryption algo, hash, lifetime). Result: IPsec SA — the actual VPN tunnel for data.
IKEv2 is the modern version — faster, more secure, supports MOBIKE (seamless handoff between networks for mobile users).
GRE tunnels
  • GRE (Generic Routing Encapsulation) creates a virtual point-to-point link (tunnel) between two routers over any IP network
  • GRE alone is NOT encrypted — it just encapsulates packets to allow multicast routing protocols (OSPF, EIGRP) over the internet
  • GRE over IPsec: GRE creates the tunnel (supports routing protocols + multicast), IPsec encrypts it. Best of both worlds.
  • Configure: interface tunnel 0 → tunnel source g0/0 → tunnel destination 203.0.113.1 → ip address 172.16.0.1 255.255.255.252
SSL VPN
  • Uses TLS (Transport Layer Security) — same encryption as HTTPS
  • Clientless SSL VPN: user opens web browser, goes to VPN portal, accesses internal web apps — no client software needed
  • Client-based SSL VPN: small client installed (Cisco AnyConnect), creates full tunnel — allows access to ALL network resources just like being in the office
  • Advantages: works through firewalls/NAT that block IPsec, easy for end users, no client needed (clientless)
Module 9 — QoS Concepts
ENSA — Module 9
Exam focus: Why QoS, QoS characteristics (bandwidth/delay/jitter/loss), QoS models (best effort/IntServ/DiffServ), classification and marking (DSCP/CoS), queuing algorithms (FIFO/WFQ/CBWFQ/LLQ), congestion avoidance.
Why QoS — the problem
  • Networks carry mixed traffic: voice (VoIP), video (conferencing), and data (files, email). Each has different requirements:
  • Voice (VoIP): needs low latency (<150ms one-way), low jitter (<30ms), low loss (<1%). Bandwidth small but time-critical.
  • Video: needs consistent bandwidth, low jitter, low loss. More tolerant of latency than voice.
  • Data: tolerant of delay, jitter, and some loss. Retransmission (TCP) handles loss.
  • When a link is congested, QoS tools decide WHICH packets to drop/delay vs which to prioritize. Without QoS = FIFO (First In First Out) — best-effort, no priorities.
QoS characteristics — 4 key metrics
CharacteristicDefinitionVoIP requirement
BandwidthAvailable capacity of a link (bps)~30-130 Kbps per call depending on codec
Delay (Latency)Time for packet to travel from source to destination<150ms one-way (recommended), <400ms maximum
JitterVariation in delay — packets arriving with inconsistent timing<30ms. Fixed by jitter buffer at receiver.
Packet lossPercentage of packets dropped<1%. Voice is UDP — no retransmission, lost = garbled audio
QoS models — three approaches
ModelHow it worksPros/Cons
Best EffortNo QoS. All packets treated equally. FIFO. First packet in = first packet out.Simple, no overhead. No guarantees — real-time apps suffer during congestion.
IntServ (Integrated Services)Applications request specific resources from EACH router along the path. Uses RSVP (Resource Reservation Protocol) to reserve bandwidth end-to-end.Guarantees are per-flow. Does NOT scale — router must track state for every flow. Only for small networks.
DiffServ (Differentiated Services)Packets are MARKED with a class (DSCP in IP header). Routers handle marked packets per-class, not per-flow. Scalable.Best scalability. No per-flow state. Most common QoS model in enterprise. Does not guarantee specific bandwidth per flow.
Classification and marking
  • Classification: identifying which traffic class a packet belongs to (by ACL, DSCP value, protocol, interface, etc.)
  • Marking: adding the class information to the packet so downstream devices can make decisions without re-classifying
  • DSCP (Differentiated Services Code Point): 6-bit field in IP header. Provides 64 possible values (0–63).
  • Common DSCP values: EF (Expedited Forwarding) = 46 — used for VoIP. AF (Assured Forwarding): AF41 (34), AF31 (26), AF21 (18), AF11 (10) — tiered data. CS values for backward compatibility.
  • CoS (Class of Service): 3-bit field in 802.1Q VLAN tag (Layer 2 marking). Range 0–7. Used within LAN. CoS 5 = voice, CoS 3 = video.
  • Trust boundary: where QoS markings from end devices are trusted. IP phones are trusted at the access switch port. PCs should NOT be trusted (users can mark their own traffic EF to get priority).
Queuing algorithms

WFQ (Weighted Fair Queuing)Automatically creates flows and provides fair bandwidth based on packet size and IP precedence. Low-volume interactive traffic gets priority.General-purpose QoS with minimal configuration
CBWFQ (Class-Based WFQ)Admin defines traffic classes (using MQC — Modular QoS CLI) and assigns minimum bandwidth guarantees to each class.When you need guaranteed minimum bandwidth per class

Hierarchical network design — 3 layers
LayerFunctionDevices
Access layerThe "edge" — connects end devices (PCs, phones, printers, APs) to the network. Port security, PoE, VLANs, STP PortFast.Access switches (Catalyst 2960, 9200 series)
Distribution layerAggregates access layer traffic. Routing between VLANs (SVIs), QoS policy enforcement, ACL filtering, route summarization. Provides redundancy to access layer.Layer 3 switches (Catalyst 9300, 9400)
Core layerHigh-speed backbone — moves large amounts of traffic between distribution blocks. Should be simple, fast, and highly available. Avoid complex policies here (slow things down).High-speed L3 switches/routers (Catalyst 9500, 9600)
Collapsed core: small networks can merge Distribution + Core into one layer (two-tier design). Saves cost.
Scalable network design principles
  • Modularity: design in blocks/modules. Each module can be upgraded independently without redesigning the whole network.
  • Hierarchy: access → distribution → core layers with clear separation of functions at each layer
  • Resiliency: no single points of failure. Redundant links, redundant devices (HSRP/VRRP), spanning tree for L2, ECMP for L3
  • Flexibility: design should accommodate growth without major redesign. VLSM for efficient IP use, modular building blocks
  • Route summarization at distribution layer: reduces number of routes core must carry. Faster convergence.
Spine-and-leaf architecture (data centers)
  • Traditional 3-tier hierarchical design doesn't scale well for modern data centers with east-west (server-to-server) traffic patterns
  • Spine-and-leaf (Clos) architecture: every leaf switch connects to every spine switch. No spine-to-spine or leaf-to-leaf connections.
  • Spine layer: high-bandwidth core. No end devices connected. Pure L3 routing between leaf switches.
  • Leaf layer: connects servers, storage, firewalls, load balancers. Uplinks to all spines.
  • Benefits: predictable latency (2 hops between any two servers), easy to add capacity (add leaf or spine), high bandwidth with ECMP load balancing across all spine uplinks
  • Uses ECMP (Equal Cost Multi-Path) routing — traffic load-balanced across all spine-to-leaf links simultaneously

SDN — Software-Defined Networking
  • Traditional networks: control plane and data plane tightly coupled inside each device. Each router/switch makes its own decisions.
  • SDN separates the control plane (WHERE to forward) from the data plane (actually forwarding the packet) into different devices.
  • Control plane → centralized SDN Controller (software on server)
  • Data plane → network devices (switches/routers) just execute forwarding decisions from controller
  • Northbound API: between controller and applications (REST API — management tools, orchestration)
  • Southbound API: between controller and network devices (OpenFlow, NETCONF)
  • OpenFlow: most common southbound API — controller tells switches exact forwarding rules (flow tables)
